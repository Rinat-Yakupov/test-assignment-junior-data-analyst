## Задание 2: FULL JOIN

**Условие:**
Представьте две таблицы: первая содержит 30 строк, а вторая — 20 строк. Мы выполняем операцию FULL JOIN между ними.

Какой диапазон возможного количества строк может быть в результирующей таблице, если учесть, что ключи для соединения могут быть как полностью совпадающими, так и абсолютно уникальными?

**Решение:**

1) Минимум - 30 строк. В этом случае все строки 2-ой таблицы соединены по ключу с какой-то одной строкой из 1-ой таблицы. Строки из 1 таблицы не могут иметь одинаковый ключ с более чем 1 строкой из второй таблицы. При этом, FULL JOIN сохраняте строки, которые не совпадают по ключу. В данном случае это еще 10 строк. И всего 20 + 10 = 30 строк.

2) Максимум - 600 строк. Этот случай получается, когда у всех строк первой таблицы и у всех строк второй один уникальный ключ для соединения. В этом случае, каждая строка 1-ой таблицы соединяется с 20 строками второй. Всего $30 \cdot 20 = 600$ строк.

**Ответ:** `минимально 30 и максимально 600 строк`

## Задание 3: Покупки

**Условие:**
```sql
create table account
(
    id integer,        -- ID счета
    client_id integer, -- ID клиента
    open_dt date,       -- дата открытия счета
    close_dt date        -- дата закрытия счета
)

create table transaction
(
    id integer,          -- ID транзакции
    account_id integer,  -- ID счета
    transaction_date date, -- дата транзакции
    amount numeric(10,2), -- сумма транзакции
    type varchar(3)       -- тип транзакции
)
```

Вывести ID клиентов, которые за последний месяц по всем своим счетам совершили покупок меньше, чем на 5000 рублей. Без использования подзапросов и оконных функций.

**Код:** [task3_purchases.sql](./task3_purchases.sql)

**Подход:** При соединении таблиц применен LEFT JOIN, чтобы сохранились клиенты, которые не сделали покупок в последнем месяце (так как $0<5000$). Фильтр по месяцу вынесен в JOIN, чтобы не отсеялись клиенты без покупок в этом месяце. Потом, клиенты группируются с помощью GROUP BY, и применяется HAVING, чтобы среди агрегированных сумм выбрать подходящие под условия.

**Допущение:**
в условии не уточняется, какие значения может принимать поле type — решение учитывает все транзакции по счёту как "покупки". Если предполагается фильтрация по конкретному типу, потребуется дополнительное условие WHERE t.type = '...'.
